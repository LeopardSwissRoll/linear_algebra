# 학습 방법론 — Claude 활용 자기주도 학습 시스템

> 이 문서는 특정 분야에 종속되지 않는 **학습 방법론**이다.
> "무엇을 배울 것인가"는 별도 커리큘럼 문서에, "어떻게 배울 것인가"는 여기에 정리한다.

---

## 1. 학습 루프의 기본 구조

모든 개념 학습은 아래 사이클을 따른다. 사이클이 짧을수록 깊이 남는다.

```
이론 습득 → 시각적 확인 → 직접 구현 → 결과 검증 → 다음 개념
   ↑                                              |
   └──────────── 막히면 여기로 돌아옴 ──────────────┘
```

### 각 단계의 원칙

**이론 습득**
- 새로운 개념을 처음 접할 때는 "왜 이것이 필요한가"부터 시작
- 정의 → 직관 → 공식 순서. 공식을 먼저 외우지 않음
- Claude에게 "이 개념을 기하학적 직관으로 설명해줘"라고 요청 가능
- 핵심 질문: "이걸 모르면 다음 단계에서 뭐가 안 되는가?"

**시각적 확인**
- 수학적 개념은 눈으로 보면 이해 속도가 배로 빨라짐
- Claude에게 인터랙티브 HTML 시각화를 요청할 수 있음 (아래 2절 참조)
- 외부 리소스(3Blue1Brown 등)와 병행

**직접 구현**
- 라이브러리 없이 핵심 연산을 직접 짜보는 단계
- "돌아가는 코드"가 아니라 "왜 이렇게 짜야 하는지 설명할 수 있는 코드"가 목표
- 이 단계를 건너뛰면 "안다"의 착각이 생김

**결과 검증**
- 직접 구현한 결과를 라이브러리/프레임워크의 결과와 수치적으로 비교
- 체크포인트 질문에 즉답할 수 있는지 자기 검증
- 즉답 불가 → 이론 단계로 돌아감

## 2. Claude 활용법 — 구체적 용도별

### 2-1. 인터랙티브 시각화 교재 제작

Claude는 단일 HTML 파일로 인터랙티브 시각화를 만들 수 있다. 요청할 때의 팁:

**요청 형식**
```
"Phase X의 [개념명]을 인터랙티브 HTML로 시각화해줘.
 - [파라미터]를 슬라이더/입력으로 조절 가능하게
 - 변환 과정을 애니메이션으로 천천히 볼 수 있게
 - MathJax로 수식도 포함
 - 3B1B 스타일로 격자/벡터가 변하는 모습"
```

**만들 수 있는 것들**
- Canvas 2D: 벡터, 행렬 변환, 좌표 격자 시각화, 내적/외적 기하학
- Three.js 3D: 3D 회전, 짐벌 락 체험, MVP 파이프라인 단계별 시각화
- MathJax 수식: 깨지지 않는 수학 표기, 실시간 값 반영
- 애니메이션: 슬라이더로 변환 진행도 제어, 자동 재생
- SVD/PCA 시각화: 단위원 → 타원 변환, 주성분 방향 표시

**현실적인 한계**
- 매우 복잡한 3D 씬(다수 메시, 조명, 그림자)은 단일 HTML로는 무거움
- WebGL 셰이더 코드는 가능하지만 디버깅이 어려울 수 있음
- 대규모 데이터셋 처리(큰 이미지, 많은 포인트)는 브라우저 성능에 제한

### 2-2. 개념 설명 및 질의응답

**효과적인 질문 패턴**
- "X를 기하학적 직관으로 설명해줘" → 공식이 아니라 그림/비유로 설명
- "X가 왜 필요한지 Y의 맥락에서 설명해줘" → 동기 부여 중심
- "X와 Y가 수학적으로 어떻게 연결되는지" → 교차점 발견
- "이 수식을 한 줄씩 해석해줘" → 수식 독해 훈련
- "X를 모르는 사람에게 설명하듯이" → 자기 이해 검증

**피해야 할 패턴**
- "X에 대해 알려줘" → 너무 넓음. 구체적으로 좁힐 것
- "이거 맞아?" → 코드/수식을 직접 보여주고 검증 요청할 것
- 개념을 듣기만 하고 구현 없이 다음으로 넘어가기

### 2-3. 코드 리뷰 및 구현 지원

**활용 방식**
- 직접 짠 코드를 보여주고 "수학적으로 올바른지 리뷰해줘"
- "이 부분에서 막혔는데, 힌트만 줘" → 정답이 아니라 방향만 요청
- 완성된 구현을 라이브러리와 비교: "DirectXMath 결과랑 내 결과가 다른데 왜?"
- 리팩토링: "이 코드를 더 수학적으로 명확하게 구조화하려면?"

**원칙**
- 막힐 때 바로 답을 요청하지 않음. 최소 30분은 스스로 시도
- 답보다 힌트를 먼저 요청
- 코드가 돌아간 후에도 "왜 이렇게 되는가"를 설명할 수 있을 때까지 질문

### 2-4. 체크포인트 검증

각 Phase 끝에서 Claude를 시험관으로 활용:

```
"Phase X를 끝냈는데, 체크포인트 질문을 내줘.
 내가 답하면 맞는지 틀린지, 부족한 부분이 뭔지 평가해줘."
```

- 즉답할 수 있으면 다음 Phase
- 설명이 모호하거나 빈 곳이 있으면 해당 개념을 다시 파기

## 3. 외부 리소스 연동 전략

### 영상 리소스

| 리소스 | 용도 | 시점 |
|--------|------|------|
| 3Blue1Brown (Essence of ~) | 기하학적 직관 형성 | 각 Phase 시작 전에 해당 챕터 시청 |
| Two Minute Papers | 최신 트렌드 감잡기 | 관련 기술이 나올 때 |
| The Cherno (C++/게임 엔진) | 실무 구현 패턴 | 구현 과제 시 참고 |

### 교재

| 교재 | 용도 | 시점 |
|------|------|------|
| 3B1B 영상 + Claude 질의 | 직관 형성 → 심화 질문 | Phase 1-2 |
| "Mathematics for 3D Game Programming" | 그래픽스 수학 레퍼런스 | Phase 3-4 참조용 |
| "Visualizing Quaternions" | 회전 심화 | Phase 4 |

### 사용 순서
1. 3B1B로 직관 잡기 (15-20분)
2. Claude에게 심화 질문 ("영상에서 X를 봤는데, 이게 Y랑 어떻게 연결돼?")
3. 직접 구현
4. 막히면 Claude에게 힌트 → 교재에서 정확한 수식 확인
5. 구현 완성 후 Claude에게 리뷰 요청

## 5. 새로운 분야에 이 방법론을 적용하는 법

이 문서의 방법론은 분야를 바꿔도 동일하게 적용 가능. 순서:

1. **커리큘럼 설계 요청**
   - Claude에게 "나의 배경은 [X]이고, [Y] 분야를 배우려는 목적은 [Z]야. 커리큘럼을 짜줘."
   - 별도 md 파일로 저장

2. **각 Phase마다 이 방법론의 학습 루프 적용**
   - 이론 → 시각화 → 구현 → 검증

3. **시각화가 필요한 개념마다 Claude에게 HTML 요청**
   - 요청 형식은 2-1절 참조

4. **Phase별 체크포인트로 진행 판단**

5. **분야 간 교차점 추적**
   - 새 분야를 배울 때 "이전 분야의 [X]와 어떻게 연결되는가?" 질문
   - 교차점이 명확해지면 두 커리큘럼을 연결하는 노트 작성

## 6. Phase 1에서 얻은 실전 교훈

> Phase 1 진행 과정에서 발견한 원칙들. 다음 Phase에도 적용할 것.

### 6-1. 수업 구조

**토픽별 개별 페이지 → 나중에 통합 고려**
- 한 페이지에 모든 토픽을 넣으면 무겁고, 개별 페이지가 집중력과 반복 학습에 유리
- 공통 CSS(`common.css`)와 캔버스 라이브러리(`canvas-lib.js`)를 분리해두면 새 페이지 제작이 매우 빨라짐

**복잡한 개념은 별도 페이지로 분리**
- "동차 좌표"처럼 여러 맥락에서 쓰이는 용어는 해당 토픽 안에 끼워넣지 말고, 전용 페이지를 만들어 깊이 있게 다루는 것이 효과적
- 용어가 "어디서나 보이지만 정확히 모르겠는" 상태가 가장 위험 → 어원 + 다중 맥락 + 인터랙티브를 한 곳에 모으면 해소됨

### 6-2. 시각화 원칙

**인터랙티브 > 정적 설명**
- 드래그 가능한 시각화가 텍스트 설명 10줄을 대체함
- 모든 수업 페이지에 최소 1개의 인터랙티브 캔버스를 포함할 것

**투영 등 비대칭 시각화는 "순서를 생각하지 않는" 방향으로**
- 내적 투영: 긴 벡터를 짧은 벡터에 사영하면 교환법칙이 직관적으로 보임
- 일반 원칙: 시각화의 기본 상태(default)가 가장 교육적인 케이스를 보여주도록 설계

**기저 변환 시 "기저 좌표 유지" 모드**
- 기저를 바꿀 때 점의 기저 좌표를 고정하고 세계 좌표를 재계산하면, "기저가 공간을 변형시킨다"는 직관이 직접적으로 보임

### 6-3. 체크포인트 = 연습 문제 + 비주얼 오답노트

**자동 채점 가능한 수학 문제**
- 난수로 매번 새로운 벡터/행렬 생성
- 수치 입력 → epsilon 허용 오차 자동 채점
- 매번 다른 문제이므로 반복 연습 가능

**비주얼 오답노트 (핵심 기능)**
- 틀린 문제에 대해 해당 Phase의 캔버스 라이브러리로 시각적 해설 생성
- 벡터, 투영, 평행사변형 등을 직접 그려서 "왜 이 답인지"를 보여줌
- 수업에서 배운 시각화를 채점에서 재활용 → 학습 루프가 닫힘

**서술형 평가는 Claude 대화로**
- "설명해봐" 류의 질문은 HTML로 채점 불가
- Phase 체크포인트의 마지막 단계로, Claude와 직접 대화하며 개념 이해도를 검증

### 6-4. 파일 구조 업데이트

```
lessons/
├── common.css              ← 공통 스타일시트
├── canvas-lib.js           ← 캔버스 유틸리티 라이브러리
├── phase1_vector_identity.html
├── phase1_dot_product.html
├── phase1_cross_product.html
├── phase1_basis.html
├── phase1_homogeneous.html
├── phase1_checkpoint.html  ← 연습 문제 + 비주얼 오답노트
└── ...
```

## 7. 메타 원칙

- **구현 없는 이해는 착각이다.** 모든 개념은 코드로 검증.
- **짧은 사이클이 긴 사이클을 이긴다.** 3시간 이론 후 구현보다, 30분 이론 → 구현 → 30분 이론 → 구현.
- **체크포인트를 건너뛰지 않는다.** 즉답 불가 = 아직 모르는 것.
- **도구(Claude, 라이브러리)는 직접 구현 후에 쓴다.** 먼저 손으로 해봐야 도구가 뭘 해주는지 보인다.
- **교차점을 항상 추적한다.** 한 분야에서 배운 수학이 다른 분야에서 어떻게 쓰이는지를 의식적으로 연결.
